前端包体积问题通常不是一天变坏的。今天加一个日期库，明天加一个图表库，后天把搜索组件全站引入。等 Lighthouse 分数掉下来，再回头拆就很麻烦。

体积预算的价值是提前设红线。它不一定要很精确，但要让团队知道：这个页面应该加载多少 JavaScript，超过多少需要解释。

## 静态页面先问需不需要 JS

内容站很多页面本来可以纯静态。文章详情、分类页、列表页，通常不需要大量客户端逻辑。能在构建时完成的，就不要放到浏览器里做。

搜索、评论、主题切换这些功能确实需要交互，但也应该按需加载。首页不应该因为某个深层页面用了图表库，就把图表库一起打进去。

## 依赖引入要看实际成本

有些依赖 API 很舒服，但包体积不小。引入前可以看一下打包结果，确认它是否支持 tree shaking，是否带了不需要的语言包或 polyfill。

如果只是格式化一个日期，原生 API 可能够用。如果需要复杂图表，再引入专门库也不迟。

## 预算要进入评审

体积预算不应该只存在性能报告里。PR 改动如果明显增加 JS，评审时就应该问：为什么需要，是否能懒加载，是否影响首屏。

性能优化最怕事后补救。把预算放在开发过程中，团队会自然形成克制。
