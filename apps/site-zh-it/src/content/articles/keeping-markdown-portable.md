Markdown 的好处是简单，但很多站点慢慢会把它变复杂：自定义组件、特殊短代码、私有容器、主题专用语法。短期看表达力强，长期看迁移困难。

我不是反对扩展语法，而是建议先问：这段内容离开当前渲染器还能不能读？如果答案是否定的，就要谨慎。

## 基础语法应该覆盖大多数需求

标题、段落、列表、代码块、引用、链接，已经能表达大部分技术内容。只有当普通 Markdown 明显不够用时，再引入扩展。

比如提示框可以用引用块表达，不一定要 `:::warning`。代码示例用标准 fenced code block，也比自定义组件更通用。

## 组件适合稳定结构

如果某种内容结构长期复用，比如 API 参数表、版本兼容矩阵、交互示例，使用组件是合理的。关键是组件语义要稳定，不能只是为了某次排版。

组件越多，内容和前端实现耦合越深。以后换框架、换主题或导出为文档时，成本会明显上升。

## 内容仓库要能被工具处理

可迁移的 Markdown 更容易做搜索、摘要、导出和质量检查。自定义语法太多，工具就需要专门适配。

技术内容会活很久。让它保持朴素，不是保守，而是给未来留余地。
