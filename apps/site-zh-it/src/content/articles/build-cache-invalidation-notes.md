构建缓存出问题时，大家第一反应常常是“缓存脏了”。但很多时候不是缓存脏，而是我们没有准确告诉工具：哪些输入会影响输出。

静态内容站尤其容易遇到这个问题。页面输出不仅受组件代码影响，也受 Markdown、站点配置、环境变量、图片资源影响。如果缓存 key 只看 `src`，却没有把内容目录算进去，就会出现文章改了但页面没变的错觉。

## 输入边界要具体

缓存系统不懂业务。它只知道你声明的文件、命令和环境变量。比如 `astro build` 的结果会受 `astro.config.mjs`、`tailwind.config.mjs`、`src/content` 和共享包影响，这些都应该进入任务输入。

对 monorepo 来说，还要注意跨包引用。应用依赖 `@tianma/ui-components`，UI 包改了以后应用也应该重建。这个依赖关系如果只写在 package manager 里，而构建系统没有看到，缓存就可能命中过头。

## 输出目录不要混用

另一个常见问题是多个任务写同一个目录。比如不同站点都输出到根目录 `dist`，本地看似方便，缓存恢复时却会互相覆盖。

每个应用应该有独立输出目录，生成物不要提交，缓存恢复后也要能被完整替换。构建产物最好是一次性生成的，不要依赖上一次残留文件。

## 失效策略要可解释

好缓存的标志不是永远命中，而是失效原因可解释。改内容重建、改布局重建、改配置重建，这些都正常。真正危险的是“不知道为什么没重建”。

遇到缓存问题时，先列出构建输入，再看输出目录，最后再怀疑工具。这个顺序能省很多时间。
