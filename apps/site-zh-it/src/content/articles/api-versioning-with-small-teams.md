API 版本管理经常被说得很重，好像不做网关、不做治理平台，就不算工程化。小团队真正需要的通常不是平台，而是一个能坚持执行的兼容规则。

我更愿意先从接口变更清单开始。每次改接口，都明确这次变更是新增字段、字段语义变化、字段删除，还是错误码调整。新增字段通常风险最小；删除字段和改变语义风险最大，必须给调用方迁移窗口。

## 版本号不是唯一工具

很多项目喜欢直接做 `/v1`、`/v2`。这当然清楚，但代价也不小：文档、测试、监控、客户端 SDK 都要跟着分叉。对于内部系统，先保证响应向后兼容，往往比马上切版本更划算。

比如新增一个 `statusText` 字段，不需要新版本。把 `status` 从数字改成字符串，就应该保留旧字段一段时间，并在文档里写清楚新字段何时成为主字段。

## 兼容窗口要有日期

“以后会删”没有意义。“2022-09-30 后删除”才有意义。兼容窗口最好写进变更记录，也要在代码里留下测试，避免旧逻辑被提前移除。

如果调用方很多，可以在服务端加简单日志，记录旧字段或旧路径的访问量。等访问量接近零，再删除会稳很多。

## 文档比抽象更重要

小团队最容易漏的是沟通。接口怎么变、谁受影响、什么时候迁移，这些如果只存在某个人脑子里，版本策略就会失效。

一个朴素的 `CHANGELOG.md`、一页接口说明、一次迁移提醒，可能比复杂网关更有效。版本管理的目标不是显得成熟，而是让调用方少踩坑。
