Astro 做内容站舒服的地方，不是它有多“新”，而是它默认不把你往复杂运行时里推。很多博客、文档站和专题站，本质上就是把内容组织好、把页面生成出来、让搜索引擎和读者都能稳定访问。Astro 的 SSG 模式正好贴合这个目标。

我会把一个 Astro 内容站拆成四层看：内容、路由、布局、构建。只要这四层边界清楚，后面文章变多、站点变多，也不容易乱。

## 内容层：不要让 Markdown 变成散装数据

早期只有三五篇文章时，直接在页面里写数组很快。但这种做法一旦扩展，维护体验会变差：标题在哪里改、摘要在哪里改、发布时间怎么排序、标签有没有写错，都会变成隐性约定。

更稳的做法是用 Astro content collections。每篇文章用 Markdown 写正文，用 frontmatter 写元数据，再在 `src/content/config.ts` 里定义 schema。比如 `title`、`description`、`tag`、`publishedAt`、`readingTime` 这些字段，看起来普通，但它们会被首页、详情页、SEO、RSS、sitemap 反复使用。

内容层最重要的是“可校验”。文章不是写完就算，构建时能发现字段缺失、日期格式错误，才适合接自动化。

## 路由层：URL 也是产品设计

文章详情页通常用 `[slug].astro`，再通过 `getStaticPaths()` 把 Markdown 文件映射成静态路径。这个机制很简单，但要早点想清楚 slug 规则。

我更倾向于文件名使用英文 slug，即使是中文站也是如此。中文 URL 不是不能用，但复制、分享、日志排查时会被编码成一长串，体验不太好。标题可以中文，路径保持稳定和可读即可。

列表页不应该知道文章来自哪里。它只负责读取 collection、排序、渲染卡片。这样以后文章来源从 Markdown 换成 CMS，列表页也不需要大改。

## 布局层：共享，但不要过度共享

站群矩阵里，共享布局很有价值。`BaseLayout` 可以统一处理 `title`、`description`、canonical、Open Graph、页头页脚等基础能力。

但共享布局不能把每个站点都变成一个模子。中文 AI 站需要公众号引流，中文 IT 站要保持干净，英文站要接 AdSense 和 GA4。这些差异应该通过 Props 控制，而不是在布局里写死站点判断。

一个比较好的边界是：布局负责页面骨架和通用 SEO；站点自己决定业务组件要不要出现。

## 构建层：本地稳定比配置漂亮更重要

理论上 Turborepo 可以管理所有站点的构建缓存，这很好。但本地环境有时会遇到包管理器 shim、Windows Shell 包装脚本、长驻进程退出异常这类问题。工程上我会保留一个简单的顺序构建脚本作为兜底。

这不是“不专业”，反而是对交付负责。CI 可以用 Turbo，本地可以用稳定脚本。只要最终每个站点都能独立输出 `dist/`，架构就是健康的。

Astro 内容站不用一开始就追求大而全。先把内容模型、URL、布局和构建跑顺，再加搜索、RSS、自动发布。内容系统搭稳之后，增长才不会变成搬砖。
