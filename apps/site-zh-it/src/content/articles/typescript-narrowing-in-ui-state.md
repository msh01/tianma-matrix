前端状态很容易写成一堆布尔值：`loading`、`error`、`empty`、`success`。一开始简单，后来就会出现互相矛盾的状态：既 loading 又 error，既 empty 又有 data。

TypeScript 的判别联合能让状态更清楚。它把“当前处于哪一种状态”写成一个明确字段，而不是让多个布尔值互相猜。

## 状态应该互斥

搜索页面可以有四种状态：`idle`、`loading`、`success`、`error`。成功状态里带结果，错误状态里带错误信息，空结果可以是 success 的一种情况，也可以单独建模。

这样组件渲染时只需要 switch `state.type`。TypeScript 会帮你收窄类型，避免在没有 data 的状态里访问 data。

## 布尔值适合独立开关

不是所有布尔值都坏。比如菜单是否打开、主题是否深色，这些是独立开关，用 boolean 很自然。

问题出在互斥流程状态上。请求生命周期、表单提交、上传进度，这些更适合用联合类型。

## 类型能表达产品状态

好的类型不只是避免报错，也能表达产品设计。有哪些状态、每个状态需要什么数据、哪些状态不可能同时出现，都可以写进类型。

前端 UI 的复杂度经常来自状态不清。把状态建模清楚，组件会少很多防御性判断。
